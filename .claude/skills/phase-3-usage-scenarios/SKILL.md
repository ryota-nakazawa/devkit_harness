# Phase 3: 利用イメージ整理

## 入力
- `projects/<slug>/phase-1-requirements.md`
- `projects/<slug>/phase-2-solutions.md`

## 処理手順

### 1. 利用シーンを書き出す
最低3つのシーンを **「誰が / いつ / どう使う / 期待結果」** のテーブルで整理する。

### 2. 役割分担
登場人物(管理者/利用者/承認者など)ごとに、何ができて何ができないかを明確化。

### 3. 操作の流れ
最も重要な1〜2シナリオについて、操作ステップを箇条書きで書き出す:
1. ユーザーが○○を開く
2. △△を入力する
3. □□が表示される

### 4. PoC 成立条件
**「ここまで動けば PoC 成功」** を明示する。これが無いと永遠に作り込みになる。

例:
- 主要シナリオ1が画面操作だけで完走できる
- ダミーデータでも見せれば客が「これで合ってる」と言える
- 異常系は対象外

## 出力

`projects/<slug>/phase-3-scenarios.md`:

```markdown
---
project: <slug>
phase: 3
created_at: <ISO8601>
---

# Phase 3: 利用イメージ

## 利用シーン
| # | 誰が | いつ | どう使う | 期待結果 |
|---|---|---|---|---|
| 1 | ... | ... | ... | ... |
| 2 | ... | ... | ... | ... |
| 3 | ... | ... | ... | ... |

## 役割分担
- **<役割A>**: できること / できないこと
- **<役割B>**: ...

## 主要シナリオの操作フロー
### シナリオ1: <名前>
1. ...
2. ...
3. ...

## PoC 成立条件
- [ ] <条件1>
- [ ] <条件2>

## スコープ外
- <あえてやらないこと>
```

## ゲートチェック
- [ ] 利用シーンが3つ以上
- [ ] 主要シナリオに操作フローがある
- [ ] PoC 成立条件が**測定可能な形**で書かれている
- [ ] スコープ外が明記されている

- Phase 4 → `in_progress`
