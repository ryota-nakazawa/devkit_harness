# Phase 8: ユーザー検証

## 入力
- `projects/<slug>/prototype/`(動くプロト)
- ユーザー(コンサル/営業)が商談で得た反応のメモ

## 役割

商談の結果を **構造化して記録** する。後のフェーズ9と learnings/ 抽出の入力になる。
これが無いと案件は学習サイクルに乗らない。

## 処理手順

### 1. 商談メモの貼り付け
ユーザーから商談メモを貰う(口頭/メール/Slack 何でも)。原文は `projects/<slug>/phase-8-meeting-notes.md` に保存。

### 2. 構造化(以下5観点)
1. **使いやすさ**: 操作で迷った点、誤解した点
2. **業務適合性**: 想定した業務フローと合っていたか
3. **不足機能**: 「これが無いと使えない」と言われたもの
4. **追加要望**: 「これがあったらいい」レベルのもの
5. **想定外の反応**: 予想していなかった食いつき/拒否

### 3. assumption 検証
phase-1 で立てた assumption と TBD のうち、商談で**確定したもの**を更新:
- assumption が正しかった → 確定マーク
- assumption が外れた → 修正内容を記録
- TBD が解決した → 内容を記録

これが**最も重要**。assumption の検証履歴がレシピ昇格の根拠になる。

## ゲートチェック
- [ ] 5観点すべてに記述あり(空でも"特になし"と明記)
- [ ] Assumption 検証テーブルがある(対象がない場合のみ省略可)
- [ ] meeting_date が ISO 形式で書かれている

## STATUS.md 更新
ゲート通過後に `projects/<slug>/STATUS.md` を更新:
- Phase 8 → `completed` + 完了日時 + 成果物パス
- Phase 9 → `in_progress`
- 「次のアクション」を Phase 9 の内容に更新
