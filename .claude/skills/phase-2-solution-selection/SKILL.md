# Phase 2: 解決手段の選定

## 入力
- `projects/<slug>/phase-1-requirements.md`

## 処理手順

### 1. プロトタイプ種別を判定

要件の性質から **プロトタイプ種別** を1つ選定する。
ユーザーに技術名は見せず、「どういう形で見せたいか」を確認する。

| 種別 | 判定基準 | 成果物の形 |
|---|---|---|
| `web-app` | 画面を見せて操作してもらう案件(予約/フォーム/ダッシュボード/LP等) | 静的 HTML |
| `gpts` | 対話型 AI エージェントが解決の核(FAQ/相談窓口/業務アシスタント等) | GPTs 設定 + システムプロンプト |
| `claude-skill` | Claude Code の skill/ワークフローが解決の核(業務自動化/レポート生成等) | SKILL.md + 関連ファイル |
| `harness` | 複数フェーズの業務プロセスをハーネス化する案件(審査フロー/オンボーディング等) | CLAUDE.md + skills 一式 |
| `bot` | Slack/LINE/Teams 等のメッセージング連携が前提の案件 | Bot 仕様書 + サンプル会話ログ |

**判定に迷う場合はユーザーに選択肢を提示する**(技術名ではなく成果物の形で説明):
```
どの形で商談に持っていきますか？
1. 画面を見せて操作してもらう（Web画面）
2. AIと対話するデモを見せる（GPTs）
3. Claude Code で業務を自動化するデモ（skill）
4. 業務プロセス全体をハーネス化する（ハーネス設計）
5. チャットツール上で動くBotのデモ（Bot）
```

### 2. 問題タイプを抽出
要件から問題タイプを1〜3個のタグで表現する(例: `booking`, `dashboard`, `chatbot-faq`, `landing`)。

### 3. recipes/ を検索
`recipes/<tag>/recipe.md` のうち問題タイプに合致するものを最大3件読み込む。
過去の `times_used` と `success_rate` を確認する。

### 4. 候補3つを比較
**選定された種別の中で**、最低3つの候補を以下の観点で比較表にする:

#### web-app の場合
| 候補 | テンプレ/構成 | 工数 | リスク | 過去レシピ |
|---|---|---|---|---|
| A | landing-page テンプレ + ダミーデータ | 半日 | 低 | recipes/booking-smb (3/3成功) |
| B | dashboard テンプレ + Chart.js | 1日 | 中 | なし |
| C | form-app テンプレ + バリデーション | 3時間 | 低 | なし |

#### gpts の場合
| 候補 | アプローチ | 工数 | リスク | 過去レシピ |
|---|---|---|---|---|
| A | FAQ特化型(Knowledge + 定型応答) | 半日 | 低 | なし |
| B | 業務アシスタント型(Actions + 外部連携mock) | 1日 | 中 | なし |
| C | 相談窓口型(長文対話 + フォローアップ) | 半日 | 低 | なし |

#### claude-skill の場合
| 候補 | skill 構成 | 工数 | リスク | 過去レシピ |
|---|---|---|---|---|
| A | 単一 skill(1入力→1出力) | 2時間 | 低 | なし |
| B | 複数 skill チェーン(前段の出力を後段が参照) | 半日 | 中 | なし |
| C | skill + MCP 連携 | 1日 | 高 | なし |

#### harness の場合
| 候補 | 構成 | 工数 | リスク | 過去レシピ |
|---|---|---|---|---|
| A | 3フェーズ構成(最小) | 半日 | 低 | なし |
| B | 5フェーズ + ゲートチェック | 1日 | 中 | なし |
| C | フル構成(skill分割 + STATUS.md + recipes) | 2日 | 中 | なし |

#### bot の場合
| 候補 | アプローチ | 工数 | リスク | 過去レシピ |
|---|---|---|---|---|
| A | 定型応答フロー(ルールベース) | 3時間 | 低 | なし |
| B | LLM 応答 + コンテキスト保持 | 半日 | 中 | なし |
| C | 外部API連携型(予約/検索等) | 1日 | 高 | なし |

**候補3つは必須**。1つだけ提案するのは禁止(比較情報が残らない)。

### 5. 選定と理由記録
ユーザーに候補を提示し、1つ選んでもらう。Claude は理由を聞かず、客観的な選定理由を生成する:

- なぜこの候補が最小実現手段なのか
- 他2候補をなぜ捨てたか(コスト/不確実性/工数)
- どの assumption が崩れたら選び直すか
