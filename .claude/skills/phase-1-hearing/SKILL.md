# Phase 1: ヒアリング

## 役割

客から事前にもらった**生のテキスト**(議事録/メール/Slack/口頭メモのテキスト化)を入力に、
構造化された `phase-1-requirements.md` を生成する。

**ゼロから5問質問する旧方式は使わない**。客の下書きを優先的に活かし、足りない部分だけ TBD として明示する。

## 入力

以下のいずれか:
- ユーザーが直接貼り付けたテキスト
- ファイルパス(`.md` `.txt` `.eml` 等)
- 「内容はこれです」と言われた会話の引用

## 処理手順

### 1. 案件 slug を決める
- ユーザーに `project_name`(英数字+ハイフン)を1問だけ確認
- 業界 + 用途で命名推奨(例: `clinic-reservation`, `cafe-membership`)
- **`projects/<slug>/` が既に存在する場合**:
  - `STATUS.md` を読み、全フェーズ完了なら「v2 を作りますか？ 新しい slug を付けてください」と案内
  - 途中フェーズで止まっている場合は「この案件の続きから再開しますか？」と確認し、再開なら `prototype-flow` の再開フローに委譲
  - **既存ディレクトリを上書きしない**
- `projects/<slug>/` を作成

### 2. STATUS.md を初期化
- `projects/<slug>/STATUS.md` を生成し、全9フェーズを `pending` で初期化
- Phase 1 を `in_progress` に設定
- フォーマットは `prototype-flow` skill の「STATUS.md — 進捗管理」を参照

### 3. 原文を保存
- `projects/<slug>/brief-original.md` に貼り付け原文をそのまま保存(改変禁止)

### 4. 構造化
以下のテンプレに沿って `projects/<slug>/phase-1-requirements.md` を生成する:

```markdown
---
project: <slug>
phase: 1
created_at: <ISO8601>
status: draft | confirmed
---

# Phase 1: 要件ヒアリング

## 背景
<客が解こうとしている課題の文脈。なぜ今これをやるのか>

## 現状業務
<現在どうやっているか。手作業/既存ツール/関係者>

## 課題
<具体的に何が困っているか。複数あれば箇条書き>

## 期待効果
<このプロトで何が解決すれば嬉しいか。定量があれば書く>

## 制約
<予算/期限/技術/法規/組織的制約>

## 利用者像
<誰が使うか。役職/IT リテラシー/人数規模>

## 未確定事項 (TBD)
- [ ] <次の商談で確認すべき論点1>
- [ ] <次の商談で確認すべき論点2>
- ...

## Claude による補足
<原文に書かれていないが推測で埋めた assumption。客に確認してもらう対象>
```

### 5. TBD 抽出ルール

以下が原文から読み取れない場合は **必ず TBD として明記**する:

| 項目 | TBD にする条件 |
|---|---|
| 利用者数 | 「数人/数十人/百人以上」のいずれにも触れていない |
| データ保管 | 期間/場所への言及なし |
| 既存システム連携 | 連携要否の明言なし |
| 成功定義 | 「どうなれば成功か」が無い |
| 予算/期限 | 数字の言及なし |

**TBD は埋めない**。推測で埋めると後で揉める。「不明」と書く。

### 6. ゲートチェック

`phase-1-requirements.md` が以下を満たしていること:

- [ ] 背景・現状業務・課題・期待効果・制約・利用者像 の6セクション全てに記述あり(空でも"明記なし"と書く)
- [ ] `## 未確定事項 (TBD)` セクションが存在する(原文が完全に揃っていれば `- なし` と明記してよい。作為的な TBD をでっち上げてはならない)
- [ ] 上記「TBD 抽出ルール」表の各項目について、原文から読み取れないものは漏れなく TBD に反映されている
- [ ] Claude による補足セクションがある
- [ ] `brief-original.md` が保存されている

満たしていない場合は次フェーズに進めない。

### 7. STATUS.md 更新
- Phase 1 を `completed` + 完了日時に更新
- Phase 2 を `in_progress` に更新
- 「次のアクション」を Phase 2 の内容に更新
- 未解決 TBD を STATUS.md に転記

### 8. ユーザーへの報告

```
✅ フェーズ1完了: 要件ヒアリング

📂 projects/<slug>/
   ├── brief-original.md
   ├── phase-1-requirements.md
   └── STATUS.md

🔍 TBD として整理した論点: <N>件
   1. <論点>
   2. <論点>
   ...

これらは次の商談で確認すべき項目です。
よろしければ「フェーズ2へ」と言ってください。修正したい点があれば指摘してください。
```

## 禁止事項

- **原文を改変しない**(`brief-original.md` は read-only 扱い)
- **推測で TBD を埋めない**(揉めの元)
- **6セクションを省略しない**(空セクションを許すと品質がばらつく)
- **`projects/<slug>/` の外に書かない**

## 次フェーズへの引き渡し

`phase-2-solution-selection` skill を呼ぶ。入力は `projects/<slug>/phase-1-requirements.md`。
