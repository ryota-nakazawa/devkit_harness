# Phase 7: プロンプト改善

## 適用条件

このフェーズは **プロトに LLM 呼び出しが含まれる場合のみ** 実行する。
チャットボット、文章生成、要約、分類などが該当。

含まれない場合はスキップして phase-8 へ。

## 入力
- `projects/<slug>/prototype/` の現状実装
- `projects/<slug>/phase-5-spec.md` の出力要件

## 処理手順

### 1. 初期プロンプトの記録
現在のプロンプトを `projects/<slug>/phase-7-prompts/v1.md` として保存。

### 2. 評価軸を定義
spec.md の「出力項目」から評価軸を抽出:
- 粒度(短すぎ/長すぎ)
- 文体(です/ます、敬語、業界用語)
- 構成(箇条書き/表/段落)
- 抜け漏れ(必須項目が出ているか)
- 誤情報(ハルシネーション)

### 3. 評価
3〜5件のテスト入力で出力を生成し、評価軸ごとに ✓/△/✗ を付ける。

### 4. 改善
弱点に応じた手段を選ぶ:
- 出力フォーマットを明示(JSON schema/テンプレ埋め込み)
- few-shot を追加(2〜3例)
- 役割設定を強化(「あなたは○○の専門家」)
- 禁止事項を明示(「〜は出力しない」)

### 5. v2 として保存
改善後を `phase-7-prompts/v2.md` に。差分理由を冒頭に書く。

### 6. 再評価
同じテスト入力で再度 ✓/△/✗ を付ける。改善があるか確認。
改善が無いなら v3 を作らず元に戻す(過剰チューニング防止)。

## 出力

`projects/<slug>/phase-7-prompts/`:

```
v1.md          # 初期版
v2.md          # 改善版 1
evaluation.md  # 評価結果(✓△✗テーブル)
```

`evaluation.md` 例:

```markdown
## テスト入力
1. <入力1>
2. <入力2>

## 評価
| バージョン | 粒度 | 文体 | 構成 | 抜け漏れ | 誤情報 |
|---|---|---|---|---|---|
| v1 | △ | ✓ | ✗ | △ | ✓ |
| v2 | ✓ | ✓ | ✓ | ✓ | ✓ |

## 採用版
v2(構成と粒度が改善)
```

## ゲートチェック
- [ ] v1 が記録されている
- [ ] 評価軸が最低3つ
- [ ] テスト入力が3件以上
- [ ] 改善があった場合のみ v2 以降を残す

## STATUS.md 更新
ゲート通過後に `projects/<slug>/STATUS.md` を更新:
- Phase 7 → `completed` + 完了日時 + 成果物パス

- 「現在のフェーズ」を `Phase 8: ユーザー検証 — waiting_for_meeting` に
